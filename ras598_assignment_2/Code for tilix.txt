Your package works like a pipeline:

planner_launch.py starts everything
planner_node.py is the main coordinator
task_client.py gets start and goal from /get_task
planner_core.py loads the map, builds the planning grid, inflates obstacles, runs A*, and prunes the path
controller.py converts the current waypoint into a Twist command
marker_helper.py builds RViz markers so you can see what the planner is doing

So the central idea is:

launch file → planner node → helper modules

class PlannerNode(Node)

Purpose

Main class
class TaskClient

4. planner_core.py
Purpose

This file contains the actual planning algorithms.

It does not deal with ROS directly.
It only deals with:

map
grid
path planning
pruning
coordinate conversion

That makes it easy to explain and debug.

Main class
class PlannerCore
Major functions
__init__(...)

Stores planning settings:

map path
map origin
original resolution
planning resolution
inflation radius
load_map_bitmap()

Loads cave_filled.png.

threshold_bitmap_to_binary(image)

Converts grayscale image into:

occupied = 1
free = 0
build_planning_grid(binary_image)

Builds a coarse occupancy grid at 0.2 m/cell.

This is the downsampling step.

inflate_obstacles(occupancy_grid)

Makes obstacles larger by 0.7 m.

This produces the inflated grid used for safe planning.

prepare_map()

Runs the whole map pipeline:

load image
threshold
build grid
inflate
world_to_grid(x, y)

Converts world coordinates into a grid cell.

Used for:

start
goal
grid_to_world(row, col)

Converts a grid cell back to world coordinates.

Used to turn path cells into waypoints.

is_in_bounds(cell, grid)

Checks if a cell is inside the map.

is_occupied(cell, grid)

Checks if cell is occupied.

is_free(cell, grid)

Checks if cell is valid and free.

heuristic(cell, goal)

Euclidean distance for A*.

get_neighbors(cell, grid)

Returns 8-connected neighbors.

reconstruct_path(came_from, goal_cell)

Builds the final path after A* finds the goal.

astar_search(start_cell, goal_cell, grid)

Runs A* on the inflated grid.

Returns raw path as grid cells.

line_of_sight(cell_a, cell_b, grid)

Checks whether two cells can be connected directly without collision.

prune_path(path_cells, grid)

Uses line-of-sight to remove unnecessary intermediate waypoints.

cells_to_world_path(path_cells)

Converts grid path into world-coordinate waypoints.

Who calls it

planner_node.py calls it in plan_mission().

Typical flow:

self.planner_core.prepare_map()
self.start_cell = self.planner_core.world_to_grid(...)
self.goal_cell = self.planner_core.world_to_grid(...)
self.raw_path_cells = self.planner_core.astar_search(...)
self.pruned_path_cells = self.planner_core.prune_path(...)
self.pruned_path_world = self.planner_core.cells_to_world_path(...)

How the files call each other

Here is the actual dependency flow:

planner_launch.py
    └── launches planner_node

planner_node.py
    ├── uses TaskClient from task_client.py
    ├── uses PlannerCore from planner_core.py
    ├── uses PathController from controller.py
    └── uses build_marker_array from marker_helper.py

So:

planner_launch.py calls planner_node.py
planner_node.py calls everything else
helper files do not call each other directly

That is a very clean modular architecture.

Full runtime flow

This is what happens when you run the package.

Step 1: Launch

You run:

ros2 launch ras598_assignment_2 planner_launch.py

This starts:

map server
lifecycle manager
grading scout
planner node
Step 2: Planner node starts

planner_node.py initializes:

publishers
subscribers
timers
helper classes

State becomes:

WAITING_FOR_TASK
Step 3: Task request

startup_step() calls TaskClient.

task_client.py:

waits for /get_task
sends Trigger request
parses response message

Then planner_node.py stores:

start_world
goal_world

State becomes:

PLANNING
Step 4: Map preparation and path planning

planner_node.py calls plan_mission().

Inside that:

planner_core.prepare_map()
world_to_grid()
astar_search()
prune_path()
cells_to_world_path()

Now the node has:

raw path
pruned path
waypoint list

If pose is not available yet:
state becomes:

WAITING_FOR_POSE

Otherwise:

TRACKING_PATH
Step 5: Pose updates

/ground_truth messages arrive.

ground_truth_callback() stores:

x
y
yaw
Step 6: Path tracking

During TRACKING_PATH:

planner_node.py chooses current waypoint
controller.py computes Twist
planner_node.py publishes /cmd_vel

At the same time:

marker_helper.py builds RViz markers
planner_node.py publishes /planner_markers
Step 7: Goal reached

When final goal tolerance is satisfied:

state becomes GOAL_REACHED
robot is stopped
markers keep showing final result

Quick summary in one paragraph

planner_launch.py starts the full assignment system. planner_node.py is the central coordinator and the only file that directly interacts with all the helper modules. It uses task_client.py to get start and goal from /get_task, uses planner_core.py to load the cave map and compute a safe pruned path, uses controller.py to convert the current waypoint into a Twist command, and uses marker_helper.py to visualize the planner state in RViz. The whole package is organized so the node acts like a state machine: get task, plan once, wait for pose if needed, track the path, then stop at the goal.
